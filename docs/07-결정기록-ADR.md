# 결정 기록 (ADR — Architecture Decision Records)

> 작성일: 2026-07-03
> 목적: 주요 설계 결정과 **근거**를 남긴다. 브레인스토밍에서 오간 트레이드오프를 보존해, 나중에·면접에서 그대로 설명할 수 있게 한다. ([06-포트폴리오-전략](06-포트폴리오-전략.md)의 "ADR" 항목)

형식: 상태 / 맥락 / 결정 / 근거 / 영향.

---

## ADR-001 — 도메인 = 1:1 로봇 축구
- **상태**: 확정
- **맥락**: 실시간 통신 + 로봇 제어를 어필할 최소 게임이 필요.
- **결정**: 11:11 풀 축구 대신 **1:1** (로봇 2 + 물리 공 + 골대 2).
- **근거**: 11:11은 복잡도 과다로 핵심(실시간 통신)이 묻힘. 1:1은 최소이면서 어필 포인트를 다 담음.
- **영향**: 군집 개념 축소 → 개별/계층 제어가 전면.

## ADR-002 — 서버 권위 시뮬레이션, Rust + rapier2d
- **상태**: 확정
- **결정**: 물리·판정은 **서버에서만**(server-authoritative). 서버 = Rust(rapier2d + tokio/axum), 클라 = 브라우저 canvas 2D.
- **근거**: 로봇 무선 제어 서사와 직결. Rust는 시스템 역량 어필 + rapier2d 품질. 사용자 C++/Go 배경으로 감당 가능.
- **영향**: 클라는 렌더만. 치트 방지·디버프 판정이 서버로 집중.

## ADR-003 — 통신: WebSocket + JSON, 30Hz 스냅샷, 보간만
- **상태**: 확정
- **결정**: WebSocket 단일 세션 · JSON · 서버 60Hz 물리 / 30Hz 브로드캐스트 · 클라 **보간(interpolation)만**. 예측/보정 없음.
- **근거**: 1:1 규모엔 보간으로 충분. "가볍게" 목표. 예측/보정은 과함.
- **영향**: 무선 시뮬레이터로 견고성 시연([02](02-네트워크-프로토콜.md) §7).

## ADR-004 — 계층형 제어
- **상태**: 확정
- **결정**: 전략 층 / 행동 층(FSM) / 액추에이터 층. 사람이 개입하는 층으로 모드(직접/전략/AI)를 구분, 런타임 전환.
- **근거**: 실제 로봇 제어 구조(플래너→컨트롤러→액추에이터)와 일치. "다양한 제어" 어필.

## ADR-005 — 파츠 커스터마이징 채택, 커머스 제외
- **상태**: 확정
- **맥락**: 로봇 다양성 심화 + "장착 부위" 아이디어. 메일/유료샵/가챠도 검토됨.
- **결정**: **파츠 조립체**(7슬롯, 결정적 스탯 기여 + 코스메틱, 해금형 로드아웃) 채택. **메일·유료샵·결제·랜덤강화(가챠)는 제외.**
- **근거**: 파츠는 로봇 테마를 살리고 코어를 심화하며 백엔드 없이 가능. 커머스는 계정·DB·영속성을 요구해 초점을 흐리고 완성 리스크↑ → 필요 시 별도 포트폴리오.
- **영향**: 로봇 = 부품 조립. 파츠 카탈로그는 개발자 배포(서버 카탈로그).

## ADR-006 — 식별/영속성 = 매직코드 최소 인증 + SQLite, **Phase 2로 분리**
- **상태**: 확정(방향), 구현은 **Phase 2**
- **맥락**: 파츠 소유·착용을 영속화하려면 사용자 식별과 저장이 필요. 익명 토큰 / 이메일 검증없음 / 매직링크 / 매직코드 / 계정 등을 검토.
- **결정**:
  - 영속성 = **SQLite**(임베디드, 파일 하나). 용도: parts 카탈로그·가격, user_parts(소유), user_loadout(착용), (선택) 재화.
  - 식별 = **패스워드리스 매직코드**(이메일로 6자리 코드 → 사용자가 입력). 링크가 아니라 **코드**라 폰/크로스디바이스에서 호스트 도달 문제 없음.
  - 인증 요구 범위 = **슬롯 참가·파츠 소유/착용에만**. **관전은 익명.**
  - **v1(코어)에는 넣지 않는다.** 코어 완성 후 **Phase 2**로 얹는다.
- **근거**: 매직코드는 비번 해싱·OAuth 없이 실제 소유 검증을 주는 최소 인증. SQLite는 인프라 부담 없이 영속성 제공. 코어(실시간 제어)가 주인공이므로 인증을 뒤로 미뤄 완성도를 지킴.
- **영향 / 필수 준수**:
  - 데이터 정책 변경: v1 인메모리 → Phase 2 SQLite 영속 + 익명→식별.
  - 보안: 코드 **시도횟수 제한 + 짧은 만료(~10분) + 1회용 + 발송 레이트리밋** 필수.
  - 발송: 개발=콘솔 출력(무료), 배포=이메일 API. base URL은 설정값(`PUBLIC_URL`)으로.
  - 재화·구매(경제)는 더 뒤로 미룸 — 우선 "소유·착용 영속화"만.

## ADR-007 — 결정성 범위 = same-build
- **상태**: 확정
- **결정**: 결정적 시뮬은 **같은 빌드·같은 머신** 범위까지만. 크로스 플랫폼 float 동일성은 추구하지 않음.
- **근거**: 크로스 플랫폼 결정성은 난이도 폭증. 리플레이/테스트엔 same-build로 충분. ([06](06-포트폴리오-전략.md) §4)

## ADR-008 — 핵심 밸런스 원칙: 승리=골, 전투=양념
- **상태**: 확정
- **결정**: **승리 조건은 골뿐**(KO 승리 없음). 전투는 목적이 아니라 **템포·공간을 버는 수단**. **제압형**(쓰러뜨리고 득점)과 **도망형**(공 몰고 회피하며 득점)이 **공존**하며 서로 하드카운터하지 않는다.
- **근거**: 실시간 통신·제어가 주인공. 전투가 축구를 잡아먹으면 정체성이 흐려짐. 3분 클락이 득점 지향을 강제.
- **영향**: 이후 모든 밸런싱의 기준. 스턴 짧게·제거 없음·태클은 공을 떨굼·회피 도구는 유효하게. 전투 피해→정지회복→템포손실이라는 **자체 밸런싱** 루프.

## ADR-009 — 전투 해결 모델
- **상태**: 확정 (수치는 튜닝)
- **결정**:
  - 로봇↔로봇 충돌만 전투 판정, 공은 데미지 없음(순수 물리).
  - 상호 데미지: `impact × (상대 attack합 / 내 defense합) × 피격부위 취약도`, 양쪽 부딪힌 부위에.
  - 효과 = 넉백/스턴/데미지, **선택은 결정적(또는 시드 고정)**, **임팩트 비례 중첩**, **세기 = effect프로필 × impact × 저항 (곱)**.
  - 부위 HP: 0 초과=점진저하+정지회복, **0=파손 다운(행동불능)→일정 시간 뒤 전체 풀 리페어**로 복귀(제거 아님).
  - 공 = **자유 물리 + 밀기 드리블**, 피격 시 공 이탈.
- **근거**: 파츠가 "숫자"를 넘어 **전투 스타일**을 정함(로드아웃=전투 정체성). 결정적 모델로 리플레이/실력 보존([ADR-007]). 파손 다운은 "득점 창"으로 전투를 축구에 환급([ADR-008]).
- **영향**: 파츠에 `attack/defense/hp/effect` 속성 추가([00 §6]). 스냅샷 `event`에 피격부위·효과 정보(이펙트용). 데미지·회복·효과선택은 **순수 함수 → TDD 최적**.
